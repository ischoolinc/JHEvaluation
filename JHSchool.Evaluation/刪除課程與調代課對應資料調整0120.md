# 刪除課程與調代課對應資料調整（0120）

## 問題描述

### 原本行為
在批次刪除課程時，若一門課程有多筆「調代課對應資料（dc_bind_key）」，會產生多筆刪除 log。

**原因分析**：
- `delete_dc_bind_key` CTE 使用 `RETURNING dc_bind_key.*`，會回傳每一筆被刪除的記錄
- `insert_dc_bind_key_log` CTE 直接從 `delete_dc_bind_key` SELECT，導致每筆被刪除的記錄都會產生一筆 log
- 當一門課程有 N 筆 `dc_bind_key` 時，就會產生 N 筆 log

**範例**：
- 課程 A 有 3 筆 `dc_bind_key` → 刪除時產生 3 筆 `刪除_課程_調代課` log
- 一次刪除 3 門課程，每門各有 2 筆 → 產生 6 筆 log（而非 3 筆）

### 問題影響
- Log 表膨脹：當課程有多筆調代課對應時，log 筆數過多
- 稽核可讀性差：難以快速了解「每門課程刪除了幾筆調代課資料」
- 與其他 log 風格不一致：`刪除_課程_課程` 是每課程 1 筆，但調代課卻是多筆

## 調整說明

### 修改位置
- **檔案**：`Program.cs`
- **方法**：`DeleteCourseWithLogSQL()`
- **SQL 片段**：CTE `insert_dc_bind_key_log`

### 修改原則
1. **不改動 `delete_dc_bind_key` 的刪除邏輯**：確保資料刪除行為不受影響
2. **僅調整 `insert_dc_bind_key_log` 的 SELECT 來源**：由「逐筆」改為「GROUP BY ref_course_id」彙總
3. **在 description 加上刪除筆數**：提升稽核可讀性

### SQL 變更

#### 修改前
```sql
,insert_dc_bind_key_log AS(
INSERT INTO log(...)
SELECT 
	...
	, delete_dc_bind_key.ref_course_id AS target_id
	, '課程調代課資料刪除，課程ID「'|| delete_dc_bind_key.ref_course_id ||'」，使用者「{1}」' AS description 
FROM
	delete_dc_bind_key
)
```

#### 修改後
```sql
,insert_dc_bind_key_log AS(
INSERT INTO log(...)
SELECT
	...
	, d.ref_course_id AS target_id
	, '課程調代課資料刪除（總結），課程ID「'
	  || d.ref_course_id
	  || '」，共刪除 '
	  || d.del_cnt
	  || ' 筆，使用者「{1}」' AS description
FROM (
	SELECT
		delete_dc_bind_key.ref_course_id,
		COUNT(*) AS del_cnt
	FROM delete_dc_bind_key
	GROUP BY delete_dc_bind_key.ref_course_id
) d
)
```

### 關鍵變更點
1. **使用子查詢 GROUP BY**：將 `delete_dc_bind_key` 依 `ref_course_id` 分組，計算每課程的刪除筆數
2. **每課程 1 筆 log**：透過 GROUP BY 確保同一課程只產生一筆 log
3. **description 包含筆數**：顯示「共刪除 N 筆」，提升稽核可讀性

## 測試結果

- **準備**：選取 1 門課程，確認該課程沒有任何調代課對應資料
- **操作**：執行刪除課程
- **預期**：不應產生 `刪除_課程_調代課` 這筆 log（因為沒有刪到 dc_bind_key）
- **測試狀態**：待測試

### Case B：單一課程，dc_bind_key 有 3 筆
- **準備**：選取 1 門課程，確保 `dc_bind_key.ref_course_id = 課程ID` 有 3 筆
- **操作**：執行刪除課程
- **預期**：
  - `action_by = '刪除_課程_調代課'` 僅 1 筆
  - description 內含 `共刪除 3 筆`
- **測試狀態**：待測試

**預期 log 範例**：
```
action_by: 刪除_課程_調代課

```

### Case C：一次刪除多門課程（每門都有多筆 dc_bind_key）
- **準備**：一次選取 3 門課程刪除，每門課 `dc_bind_key` 各有 N 筆
- **操作**：執行刪除課程
- **預期**：

  - 總筆數 = 課程數量（非 dc_bind_key 刪除總筆數）
- **測試狀態**：待測試

## 回歸確認

### 確認項目
- ✅ `delete_dc_bind_key` 的刪除行為不受影響（仍會刪除所有符合條件的記錄）
- ✅ `delete_course`、`delete_sc_attend`、`delete_sce_take` 的刪除行為不受影響
- ✅ 其他兩類 log 不受影響：
  - `刪除_學生_課程評量成績`（仍可能多筆，屬逐筆評量成績刪除）
  - `刪除_課程_課程`（每課程 1 筆）
- ✅ UI 不會因 Course cache sync 而異常（原本最後有 `Course.Instance.SyncDataBackground(...)`）

## 風險與取捨

### 風險
- **不再逐筆記錄 dc_bind_key 明細**：無法從 log 中看到每筆被刪除的 `dc_bind_key` 的詳細資訊（如 id、其他欄位）

### 取捨
- **優點**：
  - Log 筆數大幅減少，提升效能與可讀性
  - 與其他 log 風格一致（每課程 1 筆）
  - description 包含刪除筆數，仍可了解刪除規模
- **缺點**：
  - 無法從 log 追溯每筆 `dc_bind_key` 的明細
  - 若需要詳細資訊，需從資料庫備份或其他稽核機制取得

### 建議
- 若未來需要更詳細的稽核資訊，可考慮：
  1. 在 description 中加上更多資訊（如刪除的 `dc_bind_key.id` 列表）
  2. 建立獨立的稽核表記錄詳細資訊
  3. 透過資料庫 transaction log 或備份機制追溯
